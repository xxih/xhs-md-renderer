# 方案：web体验优化与配置持久化

## 1. 先重构统一配置模型，再收 Web 状态

先从 `packages/core/src/models.ts` 和 `packages/core/src/themes.ts` 扩展统一配置，明确哪些属于“非正文元素”的 chrome 配置。这里要补齐头像、认证标志以及各个身份元素的显隐字段，并把 footer 默认值切到关闭。

Web 端不能继续维护一套脱离 `RenderConfig` 的私有状态，否则预览、导出和持久化会继续分叉。

## 2. 重构页眉/页脚渲染结构

在 `packages/core/src/preview.tsx` 中把当前页眉结构改成真正可配置的模块：

- 头像支持图片源与 fallback
- 头像旁支持蓝色认证标志
- 名称、handle、日期、footer 等元素都受独立开关控制

这样 Web 与未来导出都能沿着同一渲染树工作，而不是一端先做样式拼接，另一端再补。

## 3. Web 控制面板改成中文优先的紧凑编辑器

在 `apps/web/src/App.tsx` 和 `apps/web/src/styles.css` 中重新整理左侧控制区：

- 中文标题与中文字段标签
- 更高信息密度的分组与字段布局
- 把高频配置前置，低频配置压缩
- 预留明显的导出操作区

目标不是“继续多加输入框”，而是把配置组织成能在一屏内快速理解和操作的编辑器。

## 4. 修复预览舞台定位与缩放策略

右侧预览偏移问题很可能来自当前基于 `transform: scale(...)` 的舞台布局和补偿 margin 组合。这里需要把预览容器改成更稳定的舞台模型，让卡片尺寸、缩放和滚动边界由容器统一控制，而不是靠单个元素做负向补偿。

## 5. 为 Web 增加直接导出图片包能力

Web 端导出必须建立在现有 `PageModel + RenderConfig` 之上，避免和 CLI 产物结构完全脱节。实现上优先考虑复用现有渲染链路和产物命名约定，在浏览器端补齐“生成图片 + 打包下载”的能力。

若现有 Node-only 导出能力不能直接复用，则需要抽出共享部分，再为 Web 增加浏览器侧导出适配层。

## 6. 本地持久化只收敛 Web 编辑态

头像与其他常用配置的持久化优先用浏览器本地存储完成。范围先收敛在：

- 头像配置
- 非正文元素显隐
- 常用身份与排版配置

不在本轮引入跨设备同步或账户体系。

## 7. 验证路径

验证需要覆盖四层：

1. `core + web + cli` 的类型检查与构建。
2. Web 手动验证：中文界面、布局压缩、预览不偏移。
3. Web 导出验证：能直接下载图片包，文件可打开。
4. 持久化验证：刷新页面后头像与关键配置仍能恢复。
