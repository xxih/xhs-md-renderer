# 规格说明：web体验优化与配置持久化

## 1. 背景

当前 `xhs-md-renderer` 的 Web 端已经能完成 Markdown 预览，但它仍更像工程调试面板，而不是可持续使用的图文编辑器。左侧配置面板偏松散、语言不统一，右侧预览在缩放后还有明显偏移，Web 端也还不能直接导出图片包。

同时，渲染层目前缺少头像、认证标志和更细粒度的非正文元素显隐控制，导致很多接近最终图文卡片的细节只能硬编码或暂时缺失。

因此本轮需要单独收敛一版“Web 中文优先体验 + 页面 chrome 配置 + Web 导出 + 本地持久化”任务。

## 2. 交付目标

### 2.1 Web 界面中文优先

系统必须将 Web 端的主要编辑界面改为中文优先，包括但不限于：

- 控制面板分组标题
- 字段标签
- 操作按钮文案
- 提示性说明文字

成功标志：默认打开 Web 页面时，用户不需要依赖英文理解主要编辑操作。

### 2.2 左侧配置面板压缩并易于扫读

系统必须重构左侧配置面板的信息密度和布局，使常用配置在一个常见桌面视口内尽可能可见。允许通过更紧凑的分组、折叠、双列布局或粘性操作区实现，但不能继续保持当前“大块留白 + 长滚动”的状态。

成功标志：在桌面视口下，用户可以一屏扫描大部分核心配置，而不是需要连续滚动多个屏高才能找到常用项。

### 2.3 右侧预览舞台不再明显偏移

系统必须修复右侧 page 预览的布局与缩放策略，确保预览卡片在容器内稳定居中、滚动自然、不出现明显漂移或被变换挤偏的观感问题。

成功标志：同一份内容生成多页时，各页预览卡片在舞台中的位置稳定，不再出现当前用户反馈的“大偏移”。

### 2.4 Web 端直接导出图片包

系统必须允许用户在 Web 端直接导出图片包，而不是只导出中间 JSON。导出结果至少应包含当前页面集合对应的图片文件，并且文件组织方式应与现有导出产物保持一致或清晰可理解。

成功标志：用户无需切换到 CLI，即可在 Web 中直接拿到可用的图片导出包。

### 2.5 补齐头像与认证标志

系统必须在页面页眉中支持头像与头像旁的蓝色认证标志，并让它们进入统一渲染配置，而不是作为 Web 私有视觉 hack。

成功标志：预览和导出页面中都可以显示头像与认证标志，且认证标志位置与头像/昵称关系稳定。

### 2.6 所有非正文元素可开关

系统必须为当前页面中的非正文元素提供显隐开关。至少覆盖：

- 头像
- 用户名
- 用户 ID / handle
- 蓝色认证标志
- 日期
- footer

这里的“非正文元素”不包含 Markdown 正文本身，也不包含未来封面专用元素。

成功标志：用户可以在 Web 中逐项开关这些元素，预览与导出都按同一配置生效。

### 2.7 头像配置可持久化

系统必须允许用户配置头像，并在刷新页面后保留该配置。持久化至少覆盖 Web 本地使用场景，不要求本轮接入远端存储。

成功标志：用户上传或选择头像后，刷新浏览器仍能恢复同一头像配置。

### 2.8 默认不显示 footer

系统必须将 footer 的默认状态调整为关闭。

成功标志：不做任何额外配置时，默认预览和默认导出都不显示 footer。

## 3. 非目标

以下内容不属于本轮：

- 独立封面模板系统
- 封面专属配置面板
- 多套封面样式切换
- 后端账户体系或云端同步
- 完整素材库管理

## 4. 关键约束

- 正文页面仍按 Markdown 内容渲染，不使用标题或非标题语法去承载封面配置。
- 与页眉、页脚、身份信息相关的配置必须进入统一模型，不能只停留在 `apps/web` 临时状态中。
- 浏览器端新增导出能力时，不能破坏现有 CLI 导出链路。
- 若为浏览器导出引入新的运行时依赖，必须控制在能被现有 Vite 应用稳定集成的范围内。

## 5. 验收标准

1. Web 控制面板主要交互已经中文化。
2. 左侧配置面板在桌面视口下明显更紧凑，常用配置可在更少滚动内完成。
3. 右侧多页预览舞台稳定，不再出现明显偏移。
4. Web 支持直接导出图片包。
5. 头像、认证标志和其他非正文元素都具备显隐配置。
6. 头像配置可在刷新后恢复。
7. 默认配置下 footer 关闭。
