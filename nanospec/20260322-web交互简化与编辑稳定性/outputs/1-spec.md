# 规格说明：web交互简化与编辑稳定性

## 1. 背景

当前 `xhs-md-renderer` 的 Web 端虽然已经具备可用的编辑、预览与导出能力，但主要界面仍明显带有工程调试视角。对普通用户而言，界面里最显眼的应该是 Markdown 编辑与图片导出，而不是一组需要理解内部数据结构的配置和中间产物。

本轮需求的核心不是继续“补一个配置项”，而是把 Web 端重新收口为一个更简洁、更直观、更适合连续编辑的成品界面，同时补上当前已经影响使用的编辑稳定性问题。

## 2. 交付目标

### 2.1 Web 主路径必须以图片交付为中心

系统必须把 Web 端的默认主路径收敛为“编辑 Markdown、查看预览、导出图片”。普通用户打开页面后，应该能直接理解最终产物是图片，而不是 `pages.json`。

成功标志：

- 主操作区首先呈现图片导出相关动作，而不是中间数据下载。
- 新用户无需理解 `pages.json` 的含义，也能完成一次完整导出。
- 界面文案明确表达 Web 端的目标产物是图片结果。

### 2.2 `pages.json` 不再作为一级主操作暴露

系统必须降低 `pages.json` 在界面中的显性层级。它不能继续与“导出图片包”并列为普通用户的同级主按钮。

成功标志：

- 默认界面首屏中，`pages.json` 不再是一级主操作。
- 如果仍保留 `pages.json` 导出能力，它必须被归入高级操作、调试入口或更隐蔽的位置。
- 普通用户在不接触 `pages.json` 的前提下，仍能完成主要工作流。

### 2.3 Markdown 编辑区必须成为默认主工作区

系统必须显著提升 Markdown 编辑区的视觉优先级与可编辑面积，使其成为左侧或主内容区域中最重要的操作区，而不是夹在多组配置之间的小输入框。

成功标志：

- 在常见桌面视口下，Markdown 编辑区默认即可展示足够长的连续文本，不需要先手动拉伸文本框才能开始认真编辑。
- Markdown 编辑区在默认界面中的面积和层级明显高于次要配置项。
- 用户进入页面后，可以先编辑正文，再按需展开或查看次要设置，而不是先被配置卡片包围。

### 2.4 次要配置必须降噪，不与正文编辑争抢空间

系统必须压缩“画布与排版”以及其他低频设置的默认占位，使这些配置退居次要位置，不再大面积挤占编辑空间。

这里的次要配置至少包括：

- 画布与排版
- 身份信息
- 非正文元素开关
- 日期与页脚文案

成功标志：

- 默认界面中，正文编辑区明显先于次要配置获得空间。
- 低频设置不再以多个同等权重的大卡片持续占据主要视野。
- 用户完成常规编辑与导出时，不需要频繁穿越大段配置区。

### 2.5 预览区应服务编辑决策，而不是放大控制负担

系统必须保留实时预览，但预览区的说明、元信息与舞台布局应继续围绕“帮助用户判断导出结果”这一目标，而不是制造额外理解负担。

成功标志：

- 用户能清楚看到当前生成页数和主要预览结果。
- 预览相关辅助信息保持简洁，不抢占主工作区注意力。
- 编辑、预览、导出三者之间的关系在界面上清晰可理解。

### 2.6 Markdown 编辑行为必须稳定，不得发生光标跳尾

系统必须修复当前 Markdown 编辑区的核心交互问题。用户在正文中间插入换行时，光标必须停留在插入位置附近继续编辑，不能自动跳到全文末尾。

成功标志：

- 在一段已有正文的中间按回车后，换行插入成功，光标仍停留在新行位置继续输入。
- 在文本中部连续输入、换行、删除时，编辑焦点不发生异常跳转。
- 用户无需通过回到文末再手动定位的方式继续编辑。

### 2.7 Web 端必须继续保持“直接可交付”的定位

系统必须让用户把 Web 端视为可直接交付图片结果的使用入口，而不是 CLI 的附属调试面板。

成功标志：

- Web 端的默认说明、主要操作和界面结构都围绕最终图片产物组织。
- 页面中的内部概念和工程概念被弱化到不影响普通使用的程度。
- 即使保留高级能力，普通用户也不会被迫理解内部渲染中间态。

## 3. 非目标

以下内容不属于本轮：

- 新增后端服务、账号系统或云端同步
- 重做渲染引擎或更换 Markdown 解析模型
- 扩展新的封面编辑器或封面模板系统
- 补充大量新的样式主题、字体资产或复杂排版能力
- 围绕 `pages.json` 设计新的公开用户工作流

## 4. 关键约束

- Web 端仍需复用当前仓库的共享渲染能力，不能为了交互简化而做出与导出结果脱节的两套表现。
- 图片导出能力必须继续保留并可用，不能因为界面重组而退化。
- `pages.json` 若保留，只能作为高级或内部能力存在，不能重新回到普通用户主路径。
- 本轮关注的是交互层级、编辑体验和交付路径，不提前规定具体组件拆分方案或状态管理实现。

## 5. 验收标准

1. Web 默认主流程已经明确收口到 Markdown 编辑、实时预览和图片导出。
2. `pages.json` 不再以一级主操作形式面向普通用户暴露；如保留，仅在更隐蔽的高级位置出现。
3. Markdown 编辑区默认可见面积显著大于当前版本，成为界面中的主要工作区。
4. 画布、排版及其他次要配置不再长期大面积占据主视野。
5. 预览区仍然清晰可用，但整体信息负担明显低于当前版本。
6. 在正文中部按回车后，光标不会跳到全文末尾，连续编辑保持稳定。
