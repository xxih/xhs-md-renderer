# 实施方案：web交互简化与编辑稳定性

## 1. 实施思路

本轮在现有 `apps/web` 单页编辑器基础上做界面重组，不改动共享渲染模型和导出结果语义。核心策略是保留当前“Markdown -> page models -> 预览/导出”的数据路径，只调整 Web 层的信息层级、默认布局和输入更新方式。

## 2. 交互层级重组

首先把主操作区重新组织成普通用户可理解的路径：

- 主按钮只强调图片导出。
- `pages.json` 从主工具栏移出，收进高级操作区域。
- 将说明文案改为围绕“直接编辑并导出图片”展开，弱化内部实现概念。

这样可以在不删除调试能力的前提下，让默认界面表达回到最终交付物。

## 3. 编辑区优先级提升

Markdown 编辑区改为左侧的核心工作面，提供更大的默认高度和更适合连续输入的文本样式。围绕编辑区保留必要的分页提示和状态信息，但避免再用多个同级卡片把正文输入区切碎。

低频配置则改为压缩布局，并放入折叠分组或较低视觉权重的设置区，让用户可以先写正文，再按需展开微调。

## 4. 光标稳定性修复

当前最可疑的交互缺陷来自把 `textarea` 的受控更新放进 `startTransition`。修复策略是不再对输入值本身做 transition 调度，只保留 `useDeferredValue` 作为 Markdown 解析和预览降载手段。

如果实现后仍出现 selection 异常，再补充更细粒度的 caret 保持逻辑；但第一步先用最小改动去掉明显不合适的调度方式。

## 5. 预览区收口

预览区继续承担“帮助判断最终图片结果”的职责，但压缩头部信息，避免让尺寸、缩放和主题等辅助信息压过正文编辑体验。多页预览结构和导出节点引用保持兼容，避免影响现有图片导出。

## 6. 验证策略

验证分三层：

1. 静态验证：`typecheck`、`test`、`build`。
2. 手动交互验证：启动 Web，确认主路径、折叠设置、`pages.json` 隐藏位置和 Markdown 编辑体验。
3. 针对性 bug 验证：在正文中部连续回车、输入、删除，确认光标不再跳尾。

验证结论会回写到任务资产中，再决定是否标记完成。

## 7. 发布收口

实现与验证完成后，按仓库既有发布流程收口：

1. 更新 `packages/cli/package.json` 版本号与 `CHANGELOG.md`。
2. 运行 `npm pack --dry-run -w xhs-md-renderer`，确认最终 tarball 包含最新 CLI、Web 产物和 README。
3. 提交到 `main`，打对应 `v*` tag 并推送，交给 GitHub Actions 完成 npm publish、GitHub Release 与 Pages 部署。
