# web交互简化与编辑稳定性

## 背景

当前 `apps/web` 已经具备 Markdown 编辑、实时预览和图片导出能力，但整体界面仍偏向“开发者调试面板”，而不是普通用户可直接使用的成品编辑器。

这次新增反馈集中在以下几点：

- `pages.json` 作为顶层按钮暴露在主工具栏中，普通用户很难理解它是什么，也不知道什么时候应该使用。
- Web 端的主交付物应该是图片结果，而不是要求用户理解中间产物。
- Markdown 编辑区域现在过小，不够适合长文编辑，也不够顺手。
- 画布、排版和其他次要配置占据了过多界面空间，抢走了主编辑任务的注意力。
- Markdown 编辑区存在明显交互问题：在文章中间按回车时，换行虽然插入成功，但光标会跳到文本末尾，破坏连续编辑。

## 当前现状观察

- 当前主工具栏同时暴露“导出图片包”和“下载 `pages.json`”，两者处于同一优先级。
- Markdown 内容区使用普通 `textarea`，默认高度较小，位于多个配置卡片之间。
- “画布与排版”“身份信息”“非正文元素”“日期与页脚文案”等模块都以完整卡片常驻展开，左侧控制区总体偏长。
- 预览区与控制区当前采用双栏布局，编辑区没有成为明显主区域。

## 当前目标

1. 把 Web 端收敛为面向普通用户的直接交付界面，主路径聚焦“编辑 Markdown -> 预览 -> 导出图片”。
2. 让 Markdown 编辑区成为默认主工作区，显著提升可见面积与连续编辑体验。
3. 降低画布、排版和其他次要设置的视觉与空间占用，让它们退居次要位置。
4. 明确 `pages.json` 的产品定位：不再作为普通用户的主路径交付物。
5. 把“回车后光标跳到末尾”的编辑稳定性问题纳入本轮范围，并在规格中给出明确验收口径。

## 当前约束

- 本轮进入实现、验证与发布，但仍聚焦当前已定义的 Web 交互简化与编辑稳定性范围，不扩展到新的产品面。
- 仍沿用当前仓库 `packages/core + packages/cli + apps/web` 的前端本地架构，不引入后端依赖作为前提。
- 不能因为简化 Web 交互而破坏现有导出链路和调试能力。
- `pages.json` 可以保留为内部或高级用途，但不能继续作为普通用户的一级主操作。
