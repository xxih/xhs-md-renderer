# 规格说明：样式对齐与选项补齐

## 1. 背景

当前 `xhs-md-renderer` 已经具备 Markdown 解析、分页预览和 PNG 导出链路，但默认页面视觉仍偏工程演示态，渲染配置项也明显少于参考实现 `note-to-red`。这会直接影响两个结果：

1. 当前导出图文与“小红书图文卡片”预期样式差距较大。
2. `core` 缺少足够的外观配置，导致 Web 和 CLI 难以共享更真实的创作参数。

因此本轮需要单独开一个任务，把样式和配置先做扎实。

## 2. 交付目标

### 2.1 默认视觉对齐

系统必须把当前页面卡片改造为更接近 `note-to-red` 默认主题的视觉结构，包括但不限于：

- 深色内容画布
- 更轻量的页眉信息区
- 更贴近图文卡片的标题与正文节奏
- 可选的页脚信息区

成功标志：默认主题导出的页面不再像通用演示卡片，而是明显更接近小红书图文帖的阅读视觉。

### 2.2 补齐统一配置模型

系统必须在统一 `core` 中补齐以下配置项，并允许 Web 与 CLI 共用：

- `fontFamily`
- `fontSize`
- `profile.name`
- `profile.handle`
- `profile.noteLabel`
- `profile.showDate`
- `profile.dateText`
- `profile.showFooter`
- `profile.footerLeft`
- `profile.footerRight`

成功标志：Web 与 CLI 可以通过同一份 `RenderConfig` 影响导出结果，而不是各自维护私有参数。

### 2.3 Web 配置面板补齐

Web 端必须暴露上述核心配置项，让用户可以在预览阶段完成常见样式调整。

成功标志：用户无需改代码即可在 Web 中调整标题标签、作者信息、字号、日期与页脚显示。

### 2.4 CLI 参数补齐

CLI 必须补充与核心配置对应的常用参数，至少覆盖：

- 主题
- 标题分割级别
- 字体族
- 字号
- 用户名
- 用户 ID
- 标题标签
- 日期显示与文案
- 页脚显示与左右文案

成功标志：无界面调用同样可以生成接近 Web 预览效果的图片产物。

## 3. 非目标

以下内容不属于本轮：

- 背景图上传、缩放、拖拽定位
- 头像图片上传与裁切
- 自定义主题编辑器
- 主题显隐管理
- 新增大量预设主题
- 跨端状态持久化系统

## 4. 关键约束

- 必须保持 `core` 为唯一配置模型收敛层。
- 默认样式允许参考 `note-to-red`，但不能直接依赖其 Obsidian 插件运行时。
- 现有 CLI 导出协议和 Web 预览链路不能被破坏。
- 若本轮发现配置边界继续膨胀，必须先回写 `alignment.md` 再扩范围。

## 5. 验收标准

1. `packages/core` 的默认页面样式已经从当前浅色演示风格切换为更接近参考实现的图文卡片风格。
2. `RenderConfig` 已补齐本轮约定的常用外观配置字段。
3. Web 端可以编辑这些字段，并立即反映到预览中。
4. CLI 可以通过参数传入这些字段，并影响导出结果。
5. 构建与至少一条实际导出链路验证通过。
