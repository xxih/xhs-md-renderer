# 规格说明：统一 Core 的 Markdown 转小红书图文工具

## 1. 背景

用户希望做一个工具，自动化小红书创作过程中的一部分工作。现有参考库 `note-to-red` 已证明“Markdown 转小红书图文图片”这一环成立，但它只覆盖创作链路的末端排版与导出。

基于 2026-03-16 的本地与外部调研，以及本轮 align 纠偏，当前任务不再是设计整条小红书创作工作流，而是先把最末端的“Markdown 转小红书图文”工具做扎实。未来整体目标会是一个 `script + skill` 配合的小红书创作工作流，但这一轮只定义其中的底层末端工具，并要求这个工具具备统一 `core`，可被 CLI、Web 和 Obsidian 插件复用。

## 2. 产品定位

这是一个面向个人创作者的本地优先末端渲染工具。它围绕一个统一 `core` 提供 Markdown 解析、分页建模、视觉模板应用、图片渲染和导出能力，并通过不同外壳满足不同使用方式：

- CLI：用于 agent 自动化、批处理和脚本串联。
- Web：用于可视化查看、编辑和预览。
- Obsidian 插件：用于在笔记库中直接调用和预览图文生成结果。

这个工具的直接交付物是“一组适合小红书图文帖的图片产物”和对应导出结果。它可以在未来被更大的 `script + skill` 工作流消费，但当前不负责上游选题、文案生成、标签生成或完整发布包编排。

导出结果至少包含以下产物：

- 适合图文帖的分页内容
- 1 组图文图片产物
- 1 份导出文件集合，便于被未来工作流或人工发布继续消费

## 3. 目标用户

- 已经有自己的 Markdown / Obsidian 笔记，想把内容变成小红书图文图片的创作者
- 正在搭建更大创作工作流，需要一个稳定末端渲染能力的开发者或 agent 用户
- 希望在本地完成图文生成，不把账号登录和自动发布直接交给脚本的人

## 4. 典型场景

### 场景 A：把一篇笔记转成一组图文图片

用户输入一篇结构化 Markdown 或选中一篇 Obsidian 笔记，工具自动按标题和分页规则拆分内容，套用视觉模板后导出多张小红书图文图片。

成功标志：用户可以在一次处理后拿到顺序正确、样式统一的多张图文图片，而不需要手工截图和排版。

### 场景 B：通过 CLI 被脚本或 agent 调用

用户在本地脚本、agent 或未来的 `script + skill` 工作流中调用 CLI，把一个或多个 Markdown 文件转换为标准化图片输出。

成功标志：同一份输入在无交互模式下也能稳定生成结构一致、命名一致的图片产物。

### 场景 C：在 Web 界面中人工校对和微调

用户在 Web 界面中查看 `core` 生成的分页结果，逐页调整分页内容、修改图文模板或视觉风格，再触发重新导出。

成功标志：图文的最终编辑发生在可视化界面中，但其底层数据和导出逻辑与 CLI、Obsidian 插件保持一致。

### 场景 D：在 Obsidian 中边写边预览图文效果

用户在 Obsidian 中修改笔记内容时，插件侧可以调用同一套 `core` 实时或按需更新预览，不需要复制粘贴到另一个工具里。

成功标志：Obsidian 插件只是一个外壳，底层分页与导出结果与 CLI / Web 保持一致。

## 5. 核心需求

### 5.1 素材输入

系统必须支持导入、读取或选择 Markdown / Obsidian 笔记内容，并保留标题层级、段落、列表、引用、代码块和图片引用等基本结构。

成功标志：对一篇 1000 至 3000 字的中文 Markdown，系统能够完整读取原始结构，不出现明显缺段或标题错乱。

### 5.2 统一 Core

系统必须存在一个统一 `core`，负责处理素材解析、分页建模、视觉模板应用、图片渲染和导出。CLI、Web 和 Obsidian 插件都只能调用这个 `core`，不能各自维护一套独立逻辑。

成功标志：相同输入在不同调用入口下，能够得到一致的数据结构和导出结果，不因外壳不同而出现能力分叉。

### 5.3 分页建模

系统必须把原始 Markdown 重组为可渲染的分页结构，至少支持标题切页、手动分页标记、页面顺序维护和分页结果持久化。

成功标志：对包含多级标题和分页标记的 Markdown，系统能够稳定生成页面序列，页面数量、顺序和内容可预测。

### 5.4 图文图片生成

系统必须能够把分页内容渲染成图文图片。这部分能力可以借助程序模块实现，但必须受统一 `core` 调用和管理，而不是埋在某个单独端里。

成功标志：同一份分页内容可以稳定导出为多张图片，图片数量、顺序和内容与分页数据一一对应。

### 5.5 可调视觉样式

系统必须支持复用视觉模板和身份信息，包括头像、昵称、标题风格、页脚文案、字体和主题样式。

成功标志：同一用户在不同素材之间可以直接套用既有模板，不需要逐篇重新配置基础视觉信息。

### 5.6 分页结果调整

系统必须允许用户检查并调整分页结果，包括调整页面顺序、修改单页文案、合并或拆分页，以及锁定已修改页面，避免重新生成时被覆盖。

成功标志：用户修改过的页面在后续再生成或再排版时，锁定内容不会被系统无提示重写。

### 5.7 导出结果

系统必须导出一组可被后续流程消费的结果，而不是只提供界面预览。导出物至少包含图片文件和足够描述本次导出的元信息。

成功标志：用户完成一次导出后，可以在目录或压缩包中拿到图片产物，并让未来脚本或人工流程继续使用这些结果。

### 5.8 多端一致性

CLI、Web 和 Obsidian 插件必须围绕同一份中间产物模型和同一套导出协议工作。

成功标志：任一入口生成的中间产物都可以被另一个入口继续消费，而不需要额外转换。

## 6. 非目标

以下内容不在当前 spec 的首批范围内：

- 上游选题、标题生成、正文生成、标签生成
- 完整的小红书创作工作流编排
- 自动登录或自动发布到小红书平台
- 多账号矩阵管理
- 评论采集、数据回流和效果分析
- 全平台分发
- 从链接抓取、音频转写、OCR 等复杂素材采集能力

## 7. 关键约束

- 中文内容优先，先服务简体中文创作场景。
- 本地优先，尽量避免把核心工作流建立在平台 cookie 或浏览器自动化之上。
- 用户必须始终保留人工审核与修改入口。
- 首批输入源先聚焦 Markdown / 笔记，不同时打开过多异构素材类型。
- 图文图片生成必须通过可编程模块完成，并被统一 `core` 管理。
- 多端是产品要求，但逻辑只允许在 `core` 中收敛一次，不允许三端各写一套。
- 未来的 `script + skill` 工作流只能把这个工具当作末端能力调用，不能反过来把当前工具 spec 膨胀成全流程产品。

## 8. 成功定义

如果满足以下条件，则认为 v1 达到预期：

1. 用户能够从一篇现有 Markdown 素材在 5 分钟内生成第一版小红书图文图片。
2. 系统的图文图片生成能力可以被 `core` 统一调用，并在 CLI、Web、Obsidian 三端复用。
3. 用户能够在导出前手工修改分页和视觉配置，而不是只能接受一次性生成结果。
4. 导出结果可以被未来的 `script + skill` 工作流继续消费，而不需要重复截图或重新排版。
5. 用户不需要接入平台账号，也能完成从 Markdown 到图文图片资产的主要整理工作。

## 9. 待确认事项

以下问题已记录到 `alignment.md`，在进入 plan 前需要一起定：

1. 多端交付顺序是否按 `core -> CLI -> Obsidian 插件 -> Web` 推进。
2. v1 是否只服务 Markdown / Obsidian 用户，还是一开始就支持更多输入源。
3. 导出结果的最小协议是什么，例如只要图片，还是图片加元信息文件。
