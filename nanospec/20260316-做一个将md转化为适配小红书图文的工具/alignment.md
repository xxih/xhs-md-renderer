# 对齐记录

## 2026-03-16

- [变更] 任务目录名沿用了“将 md 转化为适配小红书图文的工具”，但本轮调研后，`outputs/1-spec.md` 的推荐方向已收敛为“覆盖创作中间产物的创作装配台”，而不是只做单点的 Markdown 转图片工具。原因：用户原始需求更宽，且参考库 `note-to-red` 只覆盖创作链路中的一个环节。
- [变更] 用户进一步明确：需要一个统一的 `core`，并通过 skills 把创作步骤串成完整工作流；这个 `core` 既要能被 CLI 调用，也要能被 Web 端和 Obsidian 插件复用。因此产品形态不再是单一应用，而是“核心引擎 + 多端适配层”。
- [变更] 工作流入口已经收敛为“AI agent 打开 Obsidian 文档库后，沿 skills 编排完成整条链路”。因此 spec 需要把“知识库内工作流”当作主场景，而不是单纯的独立内容编辑器。
- [缺失] `⏳ 待确认` 多端形态的首发顺序尚未确认。当前建议按 `core -> CLI -> Obsidian 插件 -> Web` 的顺序规划，但这还未最终拍板。
- [缺失] `⏳ 待确认` 输入源优先级尚未确认。当前 spec 先按“Markdown / Obsidian 笔记”为首要输入，暂不把链接抓取、音频转写、图片 OCR 设为首批范围。
- [歧义] `⏳ 待确认` 是否包含自动发布到小红书。当前 spec 明确排除该能力，原因是现有自动发布方案通常依赖 cookie / 浏览器自动化，维护与合规风险都更高。
- [偏差] 上一版 spec 把“统一 `core`”错误扩大到了整个小红书创作工作流。用户已明确纠偏：统一 `core`` 仅指“md 转图文”这个末端工具；未来的 `script + skill` 小红书创作工作流会调用它，但不是当前这轮 spec 的交付对象。
- [变更] 当前任务重新收敛为“末端 md 转小红书图文工具”。它需要统一 `core`，并允许 CLI、Web、Obsidian 插件等多种外壳复用，但当前 spec 只定义这个工具本身，不展开上游文案生成、skills 编排系统和整条创作链路的产品设计。
- [变更] 未来的整体目标调整为“一个 script + skill 配合的小红书创作工作流”，而当前小工具是这个未来工作流的基础设施之一。受影响文件已同步从“全工作流系统”回写为“末端图文工具”。
- [变更] 用户要求将 `xhs-md-renderer` 迁移为独立仓库，并把当前 nanospec 任务一并迁入该仓库中继续推进。受影响文件中的仓库落点描述已从 `outputs/xhs-md-renderer` 调整为“仓库根目录”，避免迁移后路径失真。
