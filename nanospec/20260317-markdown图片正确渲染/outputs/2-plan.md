# 方案：markdown图片正确渲染

## 1. 先收敛共享图片模型，再区分环境解析职责

当前 `PageImageBlock` 只有 `alt` 和 `url`，足够表达“看到了图片节点”，但不足以支撑真实渲染。实现时应先在 `packages/core` 中定义图片区块的共享语义，例如：

- 原始 Markdown 图片引用
- 解析后的可渲染资源地址
- 加载状态或失败态
- 可用于布局估算的尺寸信息

随后再把“如何拿到这些信息”按环境拆开：CLI / Node 负责文件路径与远程资源读取，Web 负责浏览器可访问资源的加载与失败回退。

## 2. Node / CLI 侧补齐本地图片解析能力

CLI 是当前最需要支持本地 Markdown 文件图片的入口，因此要优先补齐以下能力：

- 以 Markdown 输入文件为基准解析相对图片路径
- 把本地文件内容转换为共享渲染链路可消费的图片源
- 对远程 URL 与 `data:` URL 走统一的图片资源归一化

这样 CLI 才能真正支持 `![图](./assets/demo.png)` 这类最常见写法，而不是只停留在解析阶段。

## 3. Web 侧按浏览器能力处理图片源

Web 编辑器不具备任意读取本地路径的能力，因此方案上不能假设“浏览器能像 CLI 一样直接读相对文件”。这里应当：

- 直接支持浏览器可访问的 URL / `data:` URL
- 为浏览器无法读取的本地路径提供明确失败态
- 尽量沿用共享图片区块样式和状态表达，避免出现 Web 私有 hack

目标不是让两端支持完全相同的输入类型，而是让两端都对自己的支持边界有稳定表现。

## 4. 重写共享图片渲染，而不是继续渲染 caption 型占位块

`packages/core/src/preview.tsx` 中的图片 block 需要改成真正的图片渲染结构：

- 使用真实图片元素
- 保持宽高比
- 约束在正文内容区内
- 失败时显示明确回退态
- 不再把 `alt` 默认渲染成可见 caption

由于 CLI PNG 导出也通过同一渲染树完成，这一步完成后预览和导出才能一起闭环。

## 5. 让布局分析消费真实图片尺寸口径

图片一旦进入真实渲染，`packages/core/src/layout.ts` 的估算逻辑也要同步改造：

- 优先使用已解析到的图片尺寸参与估算
- 无法获取尺寸时，使用稳定、保守的回退比例
- 去掉当前基于占位 caption 的高度估算

这样 `layout-report.json` 的结果才不会继续和真实页面错位。

## 6. 样例、文档与验证一起补齐

本轮至少需要新增一组图片样例 Markdown 和对应素材，覆盖：

- CLI 本地相对路径图片
- Web 可直接访问的 URL / `data:` 图片
- 图片加载失败回退

文档中要明确写清两端支持边界。验证顺序建议为：

1. `npm run typecheck`
2. `npm run build`
3. CLI 用带本地图片的 Markdown 样例导出 PNG
4. Web 手动验证 URL / `data:` 图片显示与失败回退
